# Confidentiality Guide

This site should showcase thinking, not expose proprietary work.

## Never publish

Do not publish:

- Customer names
- Employer-internal project names
- Source code from employers or customers
- Internal architecture diagrams
- Internal screenshots
- Internal API paths or schemas
- Proprietary metrics or deployment counts
- Roadmap details
- Security-sensitive implementation details
- Credentials, tokens, secrets, hostnames, or tenant identifiers
- Anything that would surprise a current or former employer, customer, or partner

## Safe framing

When describing internal or employer-related work, use generalized language:

- "Enterprise customer" instead of a customer name
- "Internal platform" instead of internal product names
- "Telemetry sources" instead of specific internal APIs
- "Large-scale environment" instead of exact confidential counts
- "Architecture concept" instead of proprietary diagrams

## Case study rule

A public case study should explain:

- The problem class
- The context at a high level
- The approach
- The tradeoffs
- The outcome type
- The lessons learned

It should not prove the implementation with private details.

## Standard confidentiality note

Use this note, or a variant of it, in any case study based on internal work:

> This case study is based on internal enterprise platform work. Customer names, implementation details, architecture specifics, metrics, screenshots, and proprietary information have been intentionally generalized or omitted.

## Red flag test

Before publishing, ask:

> If a former employer's CTO, legal team, or customer read this, would they see it as a professional generalized explanation rather than disclosure?

If the answer is not clearly yes, rewrite or remove it.
